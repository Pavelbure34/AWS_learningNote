EC2, EBS, and EFS recap

Well you need an application for users to use.

Amazon Elastic Computing Cloud(EC2)

This helps you ...
  • Hosting Web Server
  • Hosting database
  • authentication
  • everything that server can deal with
It is useful because...
  • you can make mistake and recover fast
  • fast duplication
  • so experiment it!
it needs Amazon Machine Image(AMI) for essential information like OS.
  • premade AMI
  • markets(proven!)
  • customzied AMI(not proven)
  • stored AMI(snapshots)
you can run EC2 instance with user data.
    types of instances
  • T series: effetive for web servers which user access and requests fluctuate.
  • M series: effective for average use(ex. server, computing...)
  • C series: computing intensive. effecfive for machine learning, high functional web server, and scientific modeling
  • X series: effective for database
  • R series: optimized for memory intensive application such as large scale database
  • P series: highly functional in GPU. effective for deep learning and machine learning
  • F series: good for genome research
time for payment for EC2 service...
  • on-demand instance: you pay for what you actually use
  • reserved instance: you pay for year based fee.
  • spot instance: auctioned from unused EC2 instances. Notified two minutes before it is sent away. can be very cheap
    Exclusive option
  • exlusive instance:physically separate from other user’s instances.

Elastic Block Storage(EBS)

    EBS helps

  • you have data even after EC2 instance is turned off or terminated

    Tips

  • For high availability, use two available areas
  • AWS root account has omnipotent authority!
    • So use IAM admin account.
    • Set up group for authority management since people can move among departments.
    • IAM policies only control access to AWS services.
    • IAM does not have visibility over Hyperviser.
    • Secret key pair for admin should be deleted!
    • Polcies: JSON format.
    • Resource based authority: S3 policies. or setting on resources.
      • Target:S3,S3 glacier,KMS
      • Controls required conditions, always inline authority, tasks given by specific security object.
    • Authority Authentication based:
      • Target:user,group,role
      • types:pre-made,customized, and inline(=only to this guy type) policies
      • controls task, resources, and required conditions.
    • Role:to give access to resources and services to IAM users.
      • better at security
      • give access to resources and services
      • give authority to external user, or different company
      • changeable! not incrementing. only one by one.
      • allocate roles by AWS magemet console, AWS cli, AssumeRole API, AWS Security Token Service
    • How to evaluate IAM authority
      • refuse? if yes->allowed? yes -> Allow
      • refuse? if no->NO or allowed? NO -> NO
    • Manage your roles and authority!
